= Upgrade from XP 7 to XP 8
:toc: right

React4XP 7 runs on Enonic XP 8. A React4XP 6 application runs on Enonic XP 7 and cannot run on XP 8, so moving to version 7 means upgrading the application itself from XP 7 to XP 8. This is a source-level upgrade, not an in-place version bump.

The upgrade has two parts:

. The standard *XP 7 to XP 8 application upgrade* - YAML descriptors, the new Gradle build and the code-level breaking changes.
This is the same for any XP application.
. A handful of *React4XP-specific* changes on top - dependency versions, build tooling, the new types and mounting the React4XP API.

== Recommended: Claude upgrader skill

The fastest way through the XP 7 to XP 8 part of the upgrade is the *xp-app-upgrader* skill for https://claude.com/claude-code[Claude Code^].
It inspects your application, presents an upgrade plan, runs the https://github.com/enonic/xp8migrator[xp8migrator^], rewrites the Gradle build files, applies the code-level breaking changes and validates the build - pausing for your approval before it changes anything.

Install the Enonic skills once, from Claude Code:

[source]
----
/plugin marketplace add enonic/ai-enonic-marketplace
/plugin install enonic-skills@enonic-marketplace
----

Then, from your application's project directory, ask Claude to upgrade it:

[source]
----

----

The skill covers the standard XP upgrade only - the steps in the next section document what it does, in case you prefer to upgrade by hand.

TIP: For the full XP 8 breaking-change list, see the https://developer.enonic.com/docs/xp/stable/release/upgrade[Upgrading to XP8^] guide.

When the skill has finished, complete the upgrade by applying the <<react4xp-specifics, React4XP specifics>> - these are not part of the generic skill.

== Manual XP 7 to XP 8 upgrade

=== Convert descriptors to YAML

XP 8 replaces XML component descriptors with YAML, identified by a `kind:` field.
Run the https://github.com/enonic/xp8migrator[xp8migrator^] from the project root to convert them automatically:

[source,shell]
----
curl -fsSL https://raw.githubusercontent.com/enonic/xp8migrator/main/migrator-install.sh | sh
./migrator -e overwrite
----

The migrator handles, among other things:

* `src/main/resources/application.xml` -> `enonic.yaml` (`kind: "Application"`)
* `src/main/resources/site/` -> `src/main/resources/cms/` - the whole tree, with `site.xml` split into `cms/site.yaml` and `cms/cms.yaml`, and every part, layout, page and content type converted to YAML with the appropriate `kind:`.
* Admin tools, APIs, services and webapp descriptors -> YAML.

NOTE: The migrator leaves the original `site/` folder and `application.xml` in place so you can verify the result.
Delete them once the build passes.

=== Update the build

XP 8 centers the build on the `com.enonic.xp.settings` Gradle settings plugin, which supplies the app-plugin version and the `xplibs` dependency catalog.

.settings.gradle
[source,groovy]
----
plugins {
    id 'com.enonic.xp.settings' version '4.1.0'
}

rootProject.name = projectName
----

In `build.gradle`, drop the version from the `com.enonic.xp.app` plugin (the settings plugin now supplies it) and reference XP libraries through the `xplibs` catalog:

.build.gradle
[source,groovy]
----
plugins {
    id 'maven-publish'
    id 'com.enonic.xp.app'
    id 'com.enonic.defaults' version '2.1.6'
    id 'com.github.node-gradle.node' version '7.1.0'
}

dependencies {
    include xplibs.content
    include xplibs.context
    include xplibs.portal
    include xplibs.auth

    include 'com.enonic.lib:lib-asset:2.0.1'
    include 'com.enonic.lib:lib-react4xp:7.0.0'
}
----

NOTE: XP libraries (`com.enonic.xp:lib-*`) are referenced through the `xplibs.*` catalog aliases (`lib-content` becomes `xplibs.content`, and so on).
Third-party libraries such as `lib-asset` and `lib-react4xp` keep their full coordinates.

Bump `xpVersion` in `gradle.properties`:

.gradle.properties
[source,properties]
----
xpVersion=8.0.3
----

The XP 8 app plugin requires Gradle 9. Bump the wrapper:

[source,shell]
----
./gradlew wrapper --gradle-version 9.6.1
----

=== Handle code-level breaking changes

XP 8 tightens a number of APIs.
The most common ones to look out for:

* The default context is no longer implicit - pass `branch` and `repository` explicitly.
* `delete*` renames: `deleteContent`, `deleteProject`, `deleteRepo`, `deleteJob`.
* `modify*` methods are replaced by `update*`.
* HTTP method names are uppercase (`GET`, `POST`, `DELETE`).
* In `logback.xml`, remove `<withJansi>true</withJansi>`.

See the https://developer.enonic.com/docs/xp/stable/release/upgrade[Upgrading to XP8^] guide for the complete list.

With the standard XP 8 upgrade done, complete the upgrade by applying the <<react4xp-specifics, React4XP specifics>> below.

== React4XP specifics

On top of the standard XP 8 upgrade, a React4XP application needs the following.

=== Mount the React4XP API

XP 8 requires each descriptor that renders React4XP to mount the `react4xp` API.
Add it to the `apis:` list of your site - and of any webapp or admin tool that renders React4XP:

.src/main/resources/cms/site.yaml
[source,yaml]
----
kind: "Site"
apis:
  - "asset"
  - "react4xp"
----

=== Update the dependencies

Move to the version 7 React4XP packages and the XP 8 types.

.package.json
[source,json]
----
"@enonic/react4xp": "7.0.0",
"@enonic/react-components": "^7.0.0",
"@enonic-types/lib-react4xp": "^7.0.0",
"@enonic-types/global": "^8.0.3",
----

NOTE: `@enonic-types/global` 8 brings in `@enonic-types/core`, which replaces the old `@enonic/js-utils` request type - see <<update-types, Update to the new types>> below.

In `build.gradle` (shown above), use `com.enonic.lib:lib-react4xp:7.0.0`.

=== Switch to the new build tooling

React4XP 7 changes how the application is compiled:

* Server-side TypeScript is compiled with https://tsdown.dev[tsdown^] - add `tsdown` and `@swc/core` to `devDependencies`.
React entries (`.tsx`) are still compiled by React4XP with Rspack.
* React4XP uses Rspack's native CSS support, so the old `mini-css-extract-plugin` is gone.
CSS Modules work out of the box for `*.module.css` and `*.module.scss` files.

TIP: The easiest way to get the exact, up-to-date build configuration (`tsdown.config.ts`, `react4xp.config.js`, `webpack.config.react4xp.js` and the `tsconfig.json` files) is to copy them from the https://github.com/enonic/starter-react4xp[React4XP starter^].

Update the npm scripts so the build runs tsdown (server code and client assets) and React4XP (the `.tsx` entries), with the type checks split into the React4XP and server projects:

.package.json
[source,json]
----
"scripts": {
  "build":                "tsdown",
  "build:react4xp":       "npm explore @enonic/react4xp -- npm run build:react4xp",
  "check":                "concurrently -c auto -g --timings npm:check:types:*",
  "check:types:react4xp": "npx tsc --noEmit -p src/main/resources/react4xp/tsconfig.json",
  "check:types:server":   "npx tsc --noEmit -p src/main/resources/tsconfig.json"
}
----

[[update-types]]
=== Update to the new types

React4XP 7 builds on `@enonic-types/core`.
In your controllers and processors, replace `Enonic.Xp.Http.Request` from `@enonic/js-utils/types/Request` with `Request` from `@enonic-types/core`.

== Build and deploy

Make a production build and deploy it to your sandbox:

[source,shell]
----
enonic project build
----

[source,shell]
----
enonic project deploy
----

You can also run the project in development mode, which watches for changes and automatically deploys them to the sandbox:

[source,shell]
----
enonic dev
----
